# Commit Rules

We use simple and clear commits.

Format:
type: short message

Types:
- feat -> new feature
- refactor -> improve structure
- docs -> specs and documentation
- fix -> bug fix

Rules:
- One logical change = one commit
- No "update", "changes", "fix stuff"
- Keep messages short and clear
- Focus on meaning, not code

Examples:
- docs: add functional map
- docs: add decision log
- refactor: connect specs into one system
