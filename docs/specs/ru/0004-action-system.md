# 0004 Система действий (Action System)

> **Краткое резюме модуля**
> - **Что делает:** отслеживает следующее Действие после Вердикта, побуждает пользователя действовать в течение 24 часов и позволяет уменьшить Действие при блокировке
> - **Зачем нужен:** обеспечивает подотчётность — без Системы действий Решения остаются мыслями, а не шагами
> - **Что НЕ должен делать:** не генерирует Вердикт, не управляет AI-промптом — только хранит и отслеживает Действие
> - **Конфликты:** должен согласовываться с 0002 (User Flow) и 0005 (Identity Engine And Pressure Layer)

---

## 1. Назначение

- Отслеживать следующее Действие после Вердикта.
- Побуждать пользователя действовать в течение 24 часов.
- Позволять уменьшить Действие до меньшего шага при блокировке.

## 2. Где используется

- Ответ на Решение.
- Элементы управления продолжением Действия.
- Постоянный баннер действия (PersistentActionBanner).
- Оценка профиля пользователя.

## 3. Основные объекты

- **Decision (Решение):** источник Действия.
- **ActionReminder:** сохранённая запись одного Действия.
- **ActionStatus:** текущее состояние Действия.
- **ActionResultStatus:** состояние подотчётности в истории: `done`, `not yet`, `skipped` или `overdue`.
- **BlockerCategory (Категория блокера):** причина, по которой пользователь не действовал (`fear`, `unclear`, `lazy`, `external`).
- **UserState:** запись поведения после выполнения или просрочки.
- **Pressure Layer (Слой давления):** система напоминаний и эскалации.

## 4. Логика (пошагово)

### Стандартный поток

1. Пользователь отправляет Решение (любой непустой ввод — без минимума символов).
2. Система вызывает AI. Стримит вердикт + обоснование + следующее действие немедленно.
3. После стриминга `DecisionGate` рендерится ниже ответа.
4. Система создаёт ActionReminder из действия в blueprint.
5. ActionReminder начинает со статусом `not yet`.
6. Баннер показывает Действие.
7. Пользователь нажимает «Готово».
8. Система помечает ActionStatus как `done`, обновляет оценку, список истории обновляется.
9. Если пользователь нажимает «Ещё нет», Действие остаётся активным со статусом `not yet`; список истории обновляется немедленно.
10. Если пользователь пропускает и начинает новое Решение, ActionStatus становится `skipped`, оценка уменьшается, список истории обновляется.
11. Если пользователь не действует — Слой давления эскалирует через 2ч, 12ч, 24ч.
12. При просрочке: ActionStatus становится `overdue` после применения штрафа; пользователь выбирает BlockerCategory.
13. Система вызывает `generateSmallerAction(action, category)` — на стороне клиента, без API.
14. Пользователь принимает меньшее действие через «Сделаю это сейчас».
15. Система вызывает `restartWithSmallerAction(id, action, category)` — перезаписывает напоминатель с новым 24-часовым счётчиком и статусом `not yet`.

## 5. Ключевые функции

- `generateSmallerAction(action, category)`: возвращает уменьшенную строку действия на основе BlockerCategory. Без вызова API.
- `restartWithSmallerAction(id, action, category)`: сохраняет меньшее действие поверх существующего напоминателя, сбрасывает `createdAt` и `dueAt` на новое 24-часовое окно, возвращает статус в `not yet`.
- `updateDecisionScoreOnActionCompletion()`: +5 к оценке.
- `updateDecisionScoreOnActionSkip()`: −5 к оценке.
- `updateDecisionScoreOnActionOverdue()`: −10 к оценке. Вызывается один раз за действие через флаг `overdueScorePenaltyApplied`.
- `getIdentityLabel(score)`: возвращает метку идентичности для заданной оценки (0–39 / 40–69 / 70–89 / 90–100).

## 5a. Правила оценки

| Событие   | Изменение |
|-----------|-----------|
| Готово    | +5        |
| Ещё нет   | 0         |
| Пропустил | −5        |
| Просрочка | −10       |

Оценка начинается с 50. Ограничена диапазоном 0–100. Хранится в `solveos_user_profile` (localStorage).

## 5b. Метки идентичности (на основе оценки)

| Диапазон | Метка                       |
|----------|-----------------------------|
| 0–39     | "You avoid decisions"       |
| 40–69    | "You are inconsistent"      |
| 70–89    | "You act on decisions"      |
| 90–100   | "You are highly reliable"   |

## 6. Хранимые данные

- `action`: текст Действия (может быть заменён меньшим действием после сброса).
- `decisionText`: исходный текст Решения, которое породило Действие.
- `status`: `not yet`, `done`, `skipped` или `overdue`. Старые записи могут содержать `pending` или `blocked` — трактуются как `not yet` до наступления срока.
- `createdAt`: время создания напоминателя (сбрасывается при принятии меньшего действия).
- `dueAt`: 24-часовой дедлайн (сбрасывается при принятии меньшего действия).
- `completedAt`: время выполнения Действия.
- `skippedAt`: время пропуска Действия.
- `blockerCategory`: выбранная категория при просрочке.
- `smallerAction`: текст уменьшенного действия, хранимый для истории.
- `updatedAt`: время последнего изменения.

## 7. Граничные случаи

- Нет текста Действия: не создавать напоминатель.
- Существующий ActionReminder: не дублировать его.
- Просроченное Действие: показать кнопки категорий, затем меньшее действие.
- Принято меньшее действие: перезаписать существующий напоминатель, сбросить счётчик.
- Заблокированное Действие: сохранить `blockerCategory`.
- Новое Решение заблокировано: пользователь должен завершить активное Действие или дождаться просрочки перед отправкой нового.
- `PersistentActionBanner`: не рендерить до монтирования на клиенте; все операции с `localStorage`, `window`, `Date.now()` и напоминателями выполняются после монтирования во избежание гидратационных ошибок.
- Нормальное состояние `PersistentActionBanner` (0–2ч): полностью скрыть баннер. Ничего не показывать первые 2 часа после принятия обязательства.

## 7a. История и подотчётность

- `getHistoryRecords()`: возвращает все ActionReminder (done, not yet, skipped, overdue), отсортированные от новых к старым.
- `getActionResultStatus()`: нормализует текущие и старые записи в `done`, `not yet`, `skipped` или `overdue`.
- `getActionResultTimestamp()`: выбирает временну́ю метку для поля «сколько времени прошло» в строке истории.
- `getActionMetrics()`: вычисляет `successRate` (% выполненных в последних 7 записях) и `streak` (последовательность выполненных от новых к старым).
- `formatTimeAgo(timestamp)`: преобразует ISO-метку в читаемое относительное время («2h ago», «3d ago»).
- Компонент `ActionHistory`: доступен для будущего UI, но в данный момент не рендерится в боковой панели. История действий остаётся внутренней памятью поведения до тех пор, пока UI не станет чище.
