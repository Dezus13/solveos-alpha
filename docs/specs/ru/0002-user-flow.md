# 0002 Пользовательский путь (User Flow)

> **Краткое резюме модуля**
> - **Что делает:** определяет основной путь пользователя от ввода Решения до выполнения Действия
> - **Зачем нужен:** все остальные модули должны соответствовать этому потоку — он источник истины для последовательности взаимодействий
> - **Что НЕ должен делать:** не описывает детали движка или логику ИИ — только пользовательский путь
> - **Конфликты:** должен согласовываться с 0003 (Decision Engine) и 0004 (Action System)

---

## 1. Назначение

- Определить основной путь пользователя
- Сделать цикл Решение → Действие → Выполнение понятным

## 2. Где используется

- Ввод решения на главной странице
- Сообщения консоли решений
- Результирующий экран
- Постоянный баннер действия (Persistent Action Banner)
- Журнал решений (Decision Journal)

## 3. Основные объекты

- **Decision (Решение):** ситуация, введённая пользователем
- **Verdict (Вердикт):** краткий результат Решения
- **Action (Действие):** обязательный шаг в течение 24 часов
- **ConversationThread:** локальные сообщения чата для поддержки контекста
- **UserState:** запись поведения после того, как пользователь действует или избегает Действия
- **Decision Journal (Журнал решений):** сохранённый список Решений

## 4. Петля удержания через открытые обязательства

Когда у пользователя есть активное действие (ещё не отмечено как выполненное), SolveOS применяет непрерывную открытую петлю. Это внутренний механизм давления — без уведомлений, без напоминаний. Давление исходит от самого незавершённого обязательства.

### Правила открытой петли

1. **Всегда показывай индикатор:** пока есть активное действие, что-то видно:
   - `normal` (0–2ч): очень тонкая полоска вверху страницы (приглушённый текст, точка акцента, кнопка «Готово»)
   - `pressure_2h` (2–12ч): янтарный баннер — «Ещё не готово?»
   - `pressure_12h` (12–24ч): оранжевый баннер — «Ты избегаешь этого»
   - `overdue` (24ч+): красный баннер — «Ты пропустил дедлайн. Почему?» + кнопки категорий блокера

2. **При открытии приложения с активным действием (без треда):** основная область пропускает `EmptyState` и сразу показывает `OpenCommitmentView`:
   - Показывает давление, соответствующее прошедшему времени
   - Показывает текст действия
   - Показывает оставшееся время (для состояний давления)
   - Кнопка «Готово»: помечает действие выполненным, обновляет оценку, показывает «Готово. / Следующее?»
   - Строка ввода скрыта — пользователь должен сначала взаимодействовать с открытым обязательством

3. **Завершение:**
   - Отметить как выполненное → давление сразу снимается
   - Баннер кратко показывает «Готово.» (без праздника, без языка серий)
   - Опциональная строка: «Следующее?» — предлагает новое решение без навязывания

4. **Никаких внешних уведомлений:** всё давление только внутри продукта. Система не отправляет письма, push-уведомления или напоминания вне приложения.

### Чего НЕ происходит

- Никаких анимаций праздника
- Никаких сообщений о наградах или языка достижений
- Никакого эмоционального языка серий
- Никакого принудительного UI — пользователь может закрыть приложение; давление внутреннее, не навязчивое

## 5. Основной поток (пошагово)

1. Пользователь вводит Решение (любой непустой ввод — без минимума символов)
2. Система проверяет: есть ли активное незавершённое Действие?
   - ДА → блокирует новое Решение. Показывает: «{PressureMessage} — сначала завершите предыдущее действие»
   - НЕТ → продолжает
3. Система отправляет текущий ввод плюс последние 8–12 локальных сообщений разговора в ИИ
4. Система вызывает ИИ. Стримит полный ответ немедленно: вердикт + рассуждение + следующее действие
5. Для коротких уточнений (`«почему?»`, `«а если нет денег?»`, `«объясни проще»`) ИИ разрешает тему из ConversationThread и отвечает напрямую, не перезапуская полный поток решения
6. После завершения стриминга ниже ответа появляется `DecisionGate`
7. Действие сохраняется как ActionReminder (запускается 24-часовой счётчик)
8. Баннер появляется вверху страницы
9. Пользователь отмечает «Готово» → оценка растёт, баннер исчезает

## 6. Петля выполнения (действие не выполнено)

1. Проходит время. Слой давления эскалирует:
   - 2ч: баннер становится янтарным, «Ещё не готово?»
   - 12ч: баннер становится оранжевым, «Ты избегаешь этого»
   - 24ч: баннер становится красным, «Ты пропустил дедлайн. Почему?»
2. При просрочке:
   a. Пользователь видит «Почему не выполнено?» с четырьмя кнопками категорий
   b. Пользователь выбирает категорию (страх / непонятно / нет энергии / внешний блок)
   c. Система генерирует меньшее действие (без вызова API)
   d. Пользователь видит меньшее действие и нажимает «Сделаю это сейчас»
   e. Система перезапускает тот же напоминатель с меньшим действием и новым 24-часовым счётчиком

## 7. Хранимые данные

- `decisionId`: запись, связанная с Действием
- `actionText`: Действие, показанное пользователю
- `conversationThread`: локальный список сообщений в браузере для продолжения чата
- `status`: `pending`, `done`, `blocked` или `skipped`
- `dueAt`: дедлайн Действия
- `completedAt`: время, когда Действие отмечено как выполненное
- `skippedAt`: время, когда Действие пропущено
- `blockerCategory`: категория, выбранная при просрочке
- `smallerAction`: уменьшенное действие после выбора категории

## 8. Граничные случаи

- Пустая ситуация: блокировать отправку
- Активное незавершённое Действие: блокировать новое Решение, показать сообщение давления
- ConversationThread повреждён или отсутствует: начать с пустого треда
- Новый чат: очистить локальный ConversationThread и сбросить состояние текущего ответа
- Просроченное Действие: показать кнопки категорий для уменьшения действия
- Данные отсутствуют: восстановить безопасное состояние напоминателя
