# SolveOS — Обзор проекта

> **Краткое резюме модуля**
> - **Что делает:** определяет главную систему SolveOS и показывает, как части продукта связаны между собой
> - **Зачем нужен:** единая точка входа для понимания архитектуры продукта
> - **Что НЕ должен делать:** не описывает детали каждого модуля — для этого есть отдельные спецификации
> - **Конфликты:** нет; это корневой обзорный документ

---

## 1. Назначение

- Определить главную систему SolveOS
- Показать, как продукт связывает Решения (Decisions), Действия (Actions) и их выполнение

## 2. Где используется

- Главная страница: пользователь начинает с Решения
- Консоль решений: пользователь вводит Решение
- Результирующий экран: система показывает Вердикт и Действие
- Баннер действия: система держит Действие видимым

## 3. Основные объекты

- **Decision (Решение):** ситуация пользователя, требующая чёткого Вердикта
- **Verdict (Вердикт):** короткий ответ системы о том, что делать
- **Action (Действие):** обязательный следующий шаг после Вердикта
- **UserState:** запись поведения, используемая Движком идентичности и Слоем давления
- **Decision Journal (Журнал решений):** сохранённый список прошлых Решений
- **DecisionGate:** компонент-пэйволл, скрывающий Вердикт и Действие до разблокировки. Ненастоящий платёж — тестирует готовность платить

## 4. Логика (пошагово)

1. Пользователь вводит Решение
2. Система проверяет ввод
3. Система генерирует Вердикт и Действие (полный blueprint, всегда вычисляется)
4. Система стримит частичный ответ: проблема и то, чего пользователь избегает. Вердикт и Действие пока не показываются
5. Ниже частичного ответа появляется `DecisionGate`: «Ты всё ещё избегаешь решения. Разблокируй вердикт + конкретный следующий шаг»
6. Пользователь нажимает «Разблокировать решение — €5» → дверь открывается (локальное состояние, без реального платежа)
7. Система показывает Вердикт, точное Действие и кнопку 24-часового обязательства
8. Система сохраняет Решение
9. Система отслеживает Действие
10. Система обновляет UserState

## 5. Как все части связаны

Система построена из простых связанных модулей:

- Пользователь начинает с Решения → (User Flow)
- Система анализирует его → (Decision Engine)
- Система выносит чёткий Вердикт
- Система формирует обязательное Действие → (Action System)
- Движок идентичности читает UserState → (Identity Engine And Pressure Layer)
- Слой давления реагирует на задержку или пропуск → (Identity Engine And Pressure Layer)
- Система отслеживает всё → (Data & Storage)
- Интерфейс показывает результат → (UI Structure)

Все части работают вместе, продвигая пользователя вперёд.

## 6. Почему такая структура важна

Каждая часть имеет одну чёткую роль:

- Ни одна часть не делает всё
- Нет путаницы между логикой
- Каждую часть легко улучшать

Это делает систему простой и масштабируемой.

## 7. Целевая аудитория

SolveOS создан для:
→ строителей на ранней стадии и солопренёров

Люди, которые:
- строят что-то (продукт, стартап, проект)
- застревают в решениях
- обдумывают вместо того, чтобы действовать

Основная боль:
«Я знаю, что делать, но не делаю этого.»

## 8. Позиционирование

SolveOS — не чат-ассистент.
SolveOS — инструмент давления для принятия решений.

### Что это значит в интерфейсе

- Открывающий вопрос конфронтационный: «Какое решение ты сейчас откладываешь?»
- Подтекст прямой: «Хватит обдумывать. Получи решение и следующий шаг.»
- Примеры формулируются как вопросы избегания, не как демонстрация функций:
  - «Стоит ли запустить это?»
  - «Я трачу время на эту идею впустую?»
  - «Чего я избегаю?»
- Плейсхолдер в поле ввода: «Опиши решение...» — не «Спроси SolveOS...»
- Нигде в интерфейсе нет языка ассистента

### Что это значит в системе

- Каждый ответ заканчивается обязательным Действием — не предложением
- Система давления усиливает тон по мере того, как пользователь продолжает без решения
- SolveOS не утешает. Он вынуждает к бинарному выбору: реши или прими стоимость бездействия

## 9. Принцип системы

SolveOS — не чат-бот.

Это система, которая:

- принимает Решения
- вынуждает к Действию
- отслеживает поведение
- наращивает инерцию

Цель — не думать. Цель — Действие.

## 10. Принцип интерфейса

Интерфейс минимален, не декоративен:

- UI расчищает путь к Действию
- Компоновка сфокусирована — чистая, прямая, без шума
- Поведение продукта специфично для SolveOS, не является клоном чата
- Оценка и история — внутренние системы поведения, не всегда видимые блоки UI
- Давление появляется только когда у пользователя есть незавершённое Действие

SolveOS — не клон общего чата.
SolveOS — система «решение → действие».
