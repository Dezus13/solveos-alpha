# Процесс реализации

## Родительская власть

Главнее этого документа:

- `спецификации/бизнес-контекст.md`
- `спецификации/правила-спецификаций.md`

Этот документ описывает порядок работы. Он не добавляет новые функции.

## Связанные спецификации

- `спецификации/бизнес-контекст.md`
- `спецификации/правила-спецификаций.md`
- `спецификации/AGENTS.md`
- `спецификации/решения-архитектуры/ADR-001-структура-спецификаций.md`
- `спецификации/план-следующих-этапов.md`

## Границы

Внутри: путь от идеи до реализации, проверка спецификаций, заметка реализации, валидация и коммит.

Вне: новые продуктовые требования, UI-дизайн, API-контракты и схема хранения данных.

## Что этот документ не имеет права определять

- новые возможности SolveOS
- поведение модели
- формат данных
- конкретный дизайн экранов

## Общий путь

Каждая работа проходит так:

идея → спецификация → решение → реализация → валидация

Нельзя прыгать сразу в код, если запрос меняет поведение продукта.

## Как начать задачу

1. Прочитать `спецификации/бизнес-контекст.md`.
2. Прочитать `спецификации/правила-спецификаций.md`.
3. Найти связанные feature specs.
4. Если задача касается API, данных, деплоя или архитектуры, прочитать связанные technical specs.
5. Проверить user stories только как примеры.

## Как понять, можно ли менять код

Код можно менять, если:

- поведение уже разрешено бизнес-контекстом
- есть feature spec, где описано нужное поведение
- технические ограничения не запрещают изменение
- нет открытого `requires owner decision`

Код нельзя менять, если:

- поведение есть только в user story
- задача спорит с бизнес-контекстом
- нет понятной границы изменения
- изменение требует нового источника данных, но storage spec не обновлен

## Implementation note

Перед кодом нужно создать или обновить короткую заметку реализации.

Заметка может быть:

- в конце связанной feature spec
- в конце связанной technical spec
- отдельным коротким файлом рядом со спецификацией, если изменение большое

Минимальный формат:

- задача
- связанные спецификации
- что меняем
- что не меняем
- как проверяем

Заметка должна быть короткой. Ее цель — оставить след решения, а не написать отчет.

## Как документировать реализацию

После изменения проверить:

- нужно ли обновить acceptance criteria
- не появился ли новый риск
- не поменялись ли границы функции
- не нужно ли добавить ADR

Если меняется важное архитектурное решение, нужен ADR в `спецификации/решения-архитектуры/`.

## Валидация

Минимальная проверка перед коммитом:

1. `npm run lint`
2. `npm run build`

Если проверку нельзя выполнить, это нужно явно написать в финальном сообщении и в PR/коммите, если он есть.

## Безопасный коммит

Перед коммитом:

1. Проверить `git status`.
2. Убедиться, что нет лишних runtime-файлов.
3. Убедиться, что изменение соответствует одной задаче.
4. Использовать короткое сообщение:

`docs: ...`, `feat: ...`, `fix: ...`, `refactor: ...`

Документационные изменения не должны случайно тащить runtime-код.
