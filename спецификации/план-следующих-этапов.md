# План следующих этапов

## Родительская власть

Главнее этого документа:

- `спецификации/бизнес-контекст.md`
- `спецификации/правила-спецификаций.md`

Этот документ показывает порядок работы. Он не добавляет требования сам по себе.

## Связанные спецификации

- `спецификации/процесс-реализации.md`
- `спецификации/решения-архитектуры/ADR-001-структура-спецификаций.md`
- `спецификации/отчёт-по-противоречиям.md`

## Границы

Внутри: очередность этапов и фокус каждого этапа.

Вне: детальные feature specs, UI-макеты, API-контракты и деплойные настройки.

## Что этот документ не имеет права определять

- новые функции без feature spec
- изменение runtime-кода
- сроки релиза
- коммерческие обещания

## Phase 1: specs foundation complete

Статус: завершено.

Что сделано:

- создана новая структура `спецификации/`
- описан бизнес-контекст
- описаны функции, технические ограничения и user stories
- проведен аудит противоречий
- задана иерархия власти спецификаций

## Phase 2: controlled implementation

Фокус: менять код только после проверки спецификаций.

Что нужно:

- использовать процесс `идея → спецификация → решение → реализация → валидация`
- добавлять короткие implementation notes
- не реализовывать спорное поведение без решения владельца
- держать изменения маленькими

## Phase 3: UI stabilization

Фокус: сделать текущий интерфейс стабильным и понятным.

Что проверить:

- первый запрос
- повторный пользователь
- глубокий анализ
- панель архитектуры
- пустые и ошибочные состояния

Новые UI-идеи сначала идут в feature spec.

## Phase 4: real user testing

Фокус: проверить SolveOS на реальных сценариях.

Что собрать:

- где пользователь понимает продукт
- где теряется
- какие ответы помогают действовать
- где память помогает, а где мешает
- какие части интерфейса лишние

Результаты тестов должны обновлять спецификации до изменения кода.

## Phase 5: deployment readiness

Фокус: подготовить продукт к надежному запуску.

Что проверить:

- переменные окружения
- хранение данных
- ошибки OpenAI/API
- build и lint
- поведение на Vercel
- минимальные правила приватности

Деплой не должен обещать аккаунты, платежи или кросс-девайс память, пока это не описано в спецификациях.
