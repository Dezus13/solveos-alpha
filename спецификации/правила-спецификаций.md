# Правила спецификаций

## Родительская власть

Главнее этого документа: `спецификации/бизнес-контекст.md` для продуктового смысла.

Этот документ управляет тем, как читать и писать спецификации. Он не задает продуктовые функции.

## Связанные спецификации

- `спецификации/бизнес-контекст.md`
- `спецификации/AGENTS.md`
- `спецификации/отчёт-по-противоречиям.md`

## Границы

Внутри: порядок власти, имена, правила дублирования и проверка противоречий.

Вне: функции продукта, API-контракты, UI и деплой.

## Что этот документ не имеет права определять

- новые возможности SolveOS
- поведение модели
- формат хранения данных
- пользовательские сценарии как обязательные требования

## Источник правды

Спецификации читаются сверху вниз:

1. `бизнес-контекст.md` — главная продуктовая правда.
2. `функции/*/спецификация.md` — функциональные требования.
3. `технические/*/спецификация.md` — ограничения реализации.
4. `истории-пользователя/*/спецификация.md` — примеры сценариев.

Нижний уровень не может отменить верхний. Если есть конфликт, верен верхний уровень.

## Роли документов

- бизнес-контекст отвечает на вопрос «зачем существует продукт»
- спецификация функции отвечает на вопрос «что должна делать функция»
- техническая спецификация отвечает на вопрос «как это ограничено текущей реализацией»
- история пользователя показывает пример, но не добавляет требований

## Имена

- функции называются `F001-короткое-название`
- технические документы называются `T001-короткое-название`
- истории называются `U001-короткое-название`
- главный файл внутри папки называется `спецификация.md`
- названия должны быть понятны без чтения файла

## Против дублирования

- не копировать один и тот же поток во все документы
- если правило уже есть выше, ниже на него ссылаются
- история пользователя не повторяет всю спецификацию функции
- техническая спецификация не пересказывает бизнес-контекст
- старые docs можно использовать как источник, но не переносить хаос

## Защита от противоречий

Перед изменением спецификации нужно проверить:

- не спорит ли текст с `бизнес-контекст.md`
- не задает ли история пользователя новое требование
- не меняет ли техническая спека поведение функции
- не описаны ли разные названия для одного и того же понятия
- не появился ли новый источник данных без описания в storage spec

Если конфликт ясен, исправить текст. Если конфликт спорный, написать `requires owner decision` и не реализовывать.

## Как писать новые спецификации

Каждая новая спецификация должна содержать:

- родительскую власть
- связанные спецификации
- границы
- что документ не имеет права определять
- критерии приемки, если это спецификация функции

Писать коротко. Без рекламных формулировок. Без «умных» слов там, где хватает обычных.
