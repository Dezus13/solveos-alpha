# Инструкции для работы со спецификациями

## Обязательный порядок

Перед любой реализацией:

- прочитать `спецификации/бизнес-контекст.md`
- прочитать `спецификации/правила-спецификаций.md`
- прочитать `спецификации/процесс-реализации.md`
- определить связанные спецификации функций
- если задача пересекает границы продукта, прочитать связанные технические спецификации
- сравнить запрос пользователя с существующими спецификациями
- создать или обновить короткую implementation note в релевантной спецификации или рядом с ней
- только после этого реализовывать, редактировать документацию или менять требования

## Иерархия власти

Нижестоящие документы не могут менять смысл вышестоящих:

- `бизнес-контекст.md` — главная продуктовая правда
- `функции/*/спецификация.md` — функциональная правда
- `технические/*/спецификация.md` — ограничения реализации
- `истории-пользователя/*/спецификация.md` — примеры использования

История пользователя не создает новое требование. Если поведение есть только в истории и отсутствует в спецификации функции, его нельзя реализовывать как обязательное без обновления этой функции.

## Как писать спецификации

- писать по-русски
- использовать простые слова
- описывать текущее поведение отдельно от будущих идей
- не добавлять сложность ради вида
- не дублировать старые документы целиком
- если старый документ полезен, кратко перенести смысл
- указывать родительскую власть, связанные спецификации, границы и что документ не имеет права определять

## Где что лежит

- `бизнес-контекст.md` — зачем существует SolveOS
- `правила-спецификаций.md` — как писать и читать спецификации
- `процесс-реализации.md` — как переводить задачу из идеи в проверенную реализацию
- `решения-архитектуры/` — ADR для важных решений
- `функции/` — пользовательские возможности
- `технические/` — устройство приложения и ограничения
- `истории-пользователя/` — практические сценарии
