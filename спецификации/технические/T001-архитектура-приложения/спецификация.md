# T001 — Архитектура приложения

## Текущая архитектура

SolveOS — Next.js приложение на App Router.

Главная страница загружает клиентский опыт через `DeferredHomeExperience`. Основная логика интерфейса живет в `HomeExperience`. API маршруты находятся в `app/api`. Бизнес-логика, память, профили, подсказки и типы лежат в `lib`.

## Важные файлы

- `app/page.tsx` — главная страница
- `app/layout.tsx` — общий layout
- `components/HomeExperience.tsx` — основной клиентский экран
- `components/DecisionConsole.tsx` — ввод и поток чата
- `components/dashboard/ArchitectureDashboard.tsx` — панель состояния системы
- `app/api/solve/route.ts` — главный API анализа
- `lib/types.ts` — общие типы результата
- `next.config.ts` — настройки Next.js

## Поток данных

1. Пользователь вводит запрос в UI.
2. `HomeExperience` собирает текущий ввод, историю чата и локальный контекст.
3. Клиент отправляет запрос в `/api/solve`.
4. API читает память, профиль и правила анализа.
5. Модель или fallback формирует результат.
6. API сохраняет решение.
7. UI обновляет чат, историю, панель и локальные состояния.

## Риски

- `HomeExperience.tsx` уже содержит много ответственности.
- `app/api/solve/route.ts` большой и чувствительный к случайным изменениям.
- Смешаны клиентская память, серверная память и вычисляемые показатели.
- Next.js версия новая, поэтому нельзя полагаться на старые знания без проверки документации.

## Ограничения

- не менять runtime-код ради косметики документации
- не менять структуру API без обновления связанных спецификаций
- не добавлять новый state manager без явной причины
- не переносить хранение данных без плана миграции

## Что нельзя менять случайно

- контракт `/api/solve`
- форму `DecisionBlueprint`
- ключи localStorage
- сохранение решений в `lib/memory.ts`
- клиентскую защиту от SSR/localStorage ошибок
