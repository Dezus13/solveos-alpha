# T002 — API и OpenAI

## Родительская власть

Главнее этой спецификации: `спецификации/бизнес-контекст.md` и feature specs про чат, анализ и глубокий анализ.

## Текущая архитектура

Главный маршрут анализа — `app/api/solve/route.ts`.

Он принимает запрос пользователя, историю разговора, язык и контекст. Затем собирает инструкции из модулей `lib`, вызывает движок модели и возвращает структурированный результат.

OpenAI используется через `lib/engine.ts`. Если ключа `OPENAI_API_KEY` нет или есть проблема с квотой, код содержит резервные пути.

## Поток данных

1. Клиент отправляет POST в `/api/solve`.
2. API нормализует ввод, язык и историю.
3. API читает историю решений.
4. API собирает память, профиль, давление и инструкции качества.
5. Движок вызывает модель или fallback.
6. API нормализует blueprint.
7. Решение сохраняется.
8. Клиент получает `SolveResponse`.

## Риски

- внутренние инструкции могут стать слишком длинными
- изменение промптов может резко изменить продуктовый тон
- модель может вернуть неполные или невалидные поля
- сетевые ошибки и лимиты OpenAI не должны ломать UI

## Ограничения

- не показывать пользователю внутренние инструкции
- не обещать точность модели
- не добавлять новые модели без проверки стоимости и поведения
- не менять формат ответа без обновления UI и спецификаций

## Границы

Внутри: запрос `/api/solve`, нормализация входа, вызов модели, fallback и форма ответа.

Вне: UI-композиция, история пользователя, деплой и постоянное хранение данных.

## Что эта спецификация не имеет права определять

- обещания продукта
- новые пользовательские сценарии
- визуальный вид ответа
- правила Git и деплоя

## Что нельзя менять случайно

- обработку пустого запроса
- маршрутизацию follow-up сообщений
- сохранение результата после ответа
- fallback при ошибке модели
- нормализацию `conversationHistory`
