## Простое правило

У каждой важной вещи должен быть один владелец.

Другие слои могут:

- читать данные
- показывать данные
- временно копировать данные для рендера
- вычислять производные значения

Но другие слои не должны становиться вторым независимым владельцем тех же данных.

## Короткая схема владельцев

```text
Пользовательский ввод
  -> React state текущего чата
  -> /api/solve request body

OpenAI stream
  -> временные chunks
  -> assistant message в React state
  -> localStorage conversation history

Non-streaming blueprint
  -> /api/solve JSON response
  -> latestBlueprint в React state
  -> server decision memory через saveDecision

Journal data
  -> server memory для DecisionJournal
  -> localStorage saved decisions для /journal
```

## Правила source of truth

### Single owner principle

Один тип данных должен иметь одного главного владельца.

Пример:

- текущий chat thread принадлежит `HomeExperience` React state
- сохраненная локальная история этого thread принадлежит `localStorage` ключу `solveos.conversation.v1`
- серверная история решений принадлежит `lib/memory.ts`

Если данные нужны в другом месте, их нужно читать из владельца или получать как derived copy.

### Derived vs persistent data

Persistent data — это данные, которые должны пережить перезагрузку страницы или новый запрос.

Derived data — это данные, которые можно пересчитать из persistent или runtime data.

| Тип | Значение | Пример |
| --- | --- | --- |
| Persistent | хранится как источник правды | `solveos.conversation.v1`, `data/decisions.json`, remote KV |
| Derived | пересчитывается при чтении | memory score, network score, pressure level, visible filtered journal |
| Runtime | живет только во время запроса | prompt context, OpenAI stream reader, server-local variables |
| UI-only | нужно только для экрана | modal open state, loading, active tab, error banner |

Derived data не нужно сохранять как отдельную правду, если его можно безопасно пересчитать.

### Temporary vs long-term data

Temporary data:

- input text before submit
- loading state
- streaming chunks
- current `Response` reader
- current API body
- pipeline-local instructions

Long-term data:

- local conversation history
- saved local decisions
- server decision memory
- action reminders
- user profile
- settings

Temporary data может исчезнуть после перезагрузки. Long-term data должен иметь явный storage owner.

### Runtime vs saved data

Runtime data помогает ответить сейчас.

Saved data помогает продолжить позже.

```text
Runtime truth: что сейчас происходит
Saved truth: что должно быть доступно после перезагрузки
Derived truth: что можно вычислить снова
```

Нельзя считать runtime data сохраненным, пока оно не записано в свой storage owner.

## Data ownership map

| Данные | Главный владелец | Где живет | Кто может читать | Что важно |
| --- | --- | --- | --- | --- |
| User message before submit | `DecisionConsole` | React state input | `DecisionConsole` | Не persistent. Пустой ввод не отправляется. |
| User message after submit | `HomeExperience` | `thread` React state | UI, request builder | Это текущая runtime-правда чата. |
| User message in API | `/api/solve` | `problem` in request body | API route, engine prompt builder | Копия для обработки, не владелец клиентского thread. |
| AI response streaming text | `HomeExperience` | assistant turn in `thread` | UI, local history persister | Собирается из chunks. |
| Streaming chunks | OpenAI stream / `ReadableStream` | transient stream | `/api/solve`, `HomeExperience` reader | Временные. Не сохраняются по отдельности. |
| Language state | `HomeExperience` for UI, `/api/solve` for request processing | React state and server-local `language` | UI, API, engine | Клиент определяет первым, сервер перепроверяет fallback. |
| Memory state | `lib/memory.ts` | local file in dev, remote KV when configured | API routes, memory modules, journal | Не равно local chat history. |
| Local history | `HomeExperience` persistence layer | `solveos.conversation.v1` | Home page UI | Хранит чат, не журнал решений. |
| Decision journal in side panel | `lib/memory.ts` through `/api/memory` | server decision memory | `DecisionJournal` | Показывает серверные решения. |
| Saved decisions page | `lib/savedDecisions.ts` | `solveos_saved_decisions` | `JournalView`, profile update | Отдельная browser-local коллекция. Не смешивать с server memory без явной миграции. |
| Analysis results | `/api/solve` non-streaming response | `SolveResponse.result` / `DecisionBlueprint` | `HomeExperience`, `SimulationResults` | Для глубокого анализа хранится в `latestBlueprint`. |
| UI-only state | Component that renders it | React state | Same component and children | Не писать в storage без отдельной причины. |
| Loading state | `HomeExperience` or local component | React state | UI components | Только runtime. |
| Error state | Component that owns the failed action | React state or error turn | UI | Ошибка не должна становиться memory truth. |
| Blueprint data | `/api/solve` + `lib/engine.ts` | JSON blueprint in non-streaming path | UI, memory save, outcome logger | После `saveDecision` saved blueprint belongs to memory entry. |
| Unlock data | Derived feature/state gates | computed from history/profile/blueprint | UI panels | Не отдельный источник правды. Вычисляется из владельцев данных. |
| Action reminders | `lib/actionReminders.ts` | `solveos_action_pressure_v1` | dashboard/action UI | Browser-local action truth. |
| User profile | `lib/userProfile.ts` | `solveos_user_profile` | journal, request profile context | Browser-local profile truth. |
| App settings | `lib/settings.ts`, `lib/settingsStore.ts` | `solveos.productSettings.v1`, `solveos_settings` | UI/settings | Два ключа имеют разные роли. Не добавлять третий без миграции. |

## Storage boundaries

### Browser memory

Browser memory — это данные, которые существуют только пока открыта страница.

Примеры:

- input text
- current stream reader
- local variables in handlers
- pending `fetch`

Browser memory быстро исчезает. Ее нельзя использовать как long-term source of truth.

### React state

React state управляет тем, что пользователь видит сейчас.

Примеры:

- `thread`
- `loading`
- `streaming`
- `latestBlueprint`
- `analysisError`
- `advancedOpen`
- journal filters

React state может быть копией saved data, но тогда владелец saved data остается в storage.

### localStorage

localStorage хранит browser-local данные между сессиями.

Текущие важные ключи:

| Ключ | Владелец | Назначение |
| --- | --- | --- |
| `solveos.conversation.v1` | `HomeExperience` | локальная история текущего чата |
| `solveos_saved_decisions` | `lib/savedDecisions.ts` | локальный список сохраненных решений |
| `solveos_action_pressure_v1` | `lib/actionReminders.ts` | действия и pressure state |
| `solveos_user_profile` | `lib/userProfile.ts` | локальный профиль решений |
| `solveos_settings` | `lib/settingsStore.ts` | настройки внешнего вида |
| `solveos.productSettings.v1` | `lib/settings.ts` | product settings и legacy appearance |

localStorage доступен только в браузере. Сервер не должен считать его своей памятью.

### Server runtime

Server runtime живет внутри одного API-запроса или server process.

Примеры:

- parsed request body
- normalized language
- conversation context
- memory instructions
- orchestration instructions
- calibration values

Server runtime может читать persistent memory и вернуть результат, но сам по себе не является durable storage.

### OpenAI response stream

OpenAI stream — это временный поток текста.

Он владеет только chunks, пока они идут. После чтения chunks:

- UI собирает assistant response
- stream закрывается
- chunks отдельно не хранятся

OpenAI stream не должен быть источником правды для журнала, localStorage или memory после завершения чтения.

```text
    |

    v
[OpenAI stream]
    |
    v
[Assistant turn in React state]
    |
    v
[localStorage: solveos.conversation.v1]
```

## Диаграмма: persistent owners

```text
Browser localStorage
  |-- solveos.conversation.v1      -> local chat history
  |-- solveos_saved_decisions      -> saved decisions page
  |-- solveos_action_pressure_v1   -> action reminders
  |-- solveos_user_profile         -> user profile
  |-- solveos_settings             -> app appearance
  |-- solveos.productSettings.v1   -> product settings

Server memory
  |-- lib/memory.ts
      |-- data/decisions.json in local development
      |-- remote KV when configured
      |-- /api/memory reads
      |-- /api/solve non-streaming saveDecision writes
```

## Data lifecycle examples

### Example 1: main chat streaming answer

1. User types a message in `DecisionConsole`.
2. Before submit, the input belongs to `DecisionConsole`.
3. On submit, `HomeExperience` adds a user turn to `thread`.
4. `HomeExperience` sends a copy to `/api/solve`.
5. `/api/solve` builds runtime context and calls `streamingSolveDecision`.
6. OpenAI returns chunks.
7. `HomeExperience` turns chunks into one assistant message.
8. `thread` changes.
9. `persistConversation` writes compact thread to `solveos.conversation.v1`.

Source of truth result:

- current visible chat: React `thread`
- saved local chat: `solveos.conversation.v1`
- chunks: discarded after assembly

### Example 2: deep analysis result

1. User opens deep analysis.
2. `HomeExperience` sends non-streaming `/api/solve` request.
3. `/api/solve` generates a `DecisionBlueprint`.
4. `/api/solve` normalizes and saves the decision through `saveDecision`.
5. API returns `SolveResponse`.
6. `HomeExperience` stores `result` in `latestBlueprint`.
7. `SimulationResults` renders from `latestBlueprint`.

Source of truth result:

- current rendered deep analysis: `latestBlueprint`
- saved decision memory: `lib/memory.ts` entry
- derived display data: `SimulationResults`

### Example 3: journal view

1. `DecisionJournal` fetches `/api/memory`.
2. `/api/memory` reads server decision memory.
3. `DecisionJournal` filters and renders entries.
4. The filtered list is derived UI state.

Source of truth result:

- saved decision history: server memory
- visible filtered journal: derived UI view

### Example 4: local saved decisions page

1. `JournalView` reads `getSavedDecisions`.
2. `getSavedDecisions` reads `solveos_saved_decisions`.
3. User marks a local saved decision as worked or failed.
4. `updateDecision` writes the updated record back to `solveos_saved_decisions`.
5. Profile update writes profile data to `solveos_user_profile`.

Source of truth result:

- local saved decision: `solveos_saved_decisions`
- local user profile: `solveos_user_profile`
- visible list/filter: React state derived from localStorage

## Safe vs unsafe examples

| Safe | Unsafe |
| --- | --- |
| Read `thread` and derive `conversationHistory` for a request. | Treat `conversationHistory` as a second saved chat history. |
| Save compact chat turns only to `solveos.conversation.v1`. | Save the same chat thread to multiple localStorage keys. |
| Read server decisions through `/api/memory` for `DecisionJournal`. | Let `DecisionJournal` mutate server memory directly from UI state. |
| Recompute memory score from decision history. | Store memory score as permanent truth and let it drift from history. |
| Keep loading and error flags in React state. | Persist loading/error flags as if they were product data. |
| Use OpenAI chunks to build one assistant message. | Save every chunk separately as a permanent answer. |
| Let `/api/solve` normalize language for the current request. | Let old memory override the language of the current user message. |
| Store local profile in `solveos_user_profile`. | Store profile copies inside every chat message. |
| Treat blueprint from API as immutable display input after response. | Let UI rewrite blueprint fields and then pretend it came from the engine. |

## What must NEVER happen

- duplicated ownership for the same data
- two persistence locations for the same truth without a migration plan
- UI mutating source data directly when an owner API/helper exists
- memory overriding the current user message
- memory overriding the language of the current message
- local chat history being treated as server decision memory
- server memory being treated as browser localStorage
- OpenAI chunks being saved as separate permanent records
- derived scores being saved as independent truth when they can be recalculated
- loading/error state being mixed into saved decision history
- `DecisionBlueprint` shape changing without updating `T002`, `F003` and UI consumers
- localStorage keys changing without updating `T003`
- journal data being silently merged across server memory and browser saved decisions
- runtime prompt/context being shown as user-visible data

## Practical owner rules

Use these questions before changing data flow:

1. Does this data need to survive reload?
2. If yes, which existing storage owner already owns it?
3. If no, should it stay in React state or server runtime only?
4. Can this value be derived from another source?
5. If it can be derived, why would we persist it?
6. If two places store it, which one wins after conflict?

If the answer to question 6 is unclear, the change needs a spec update before implementation.
