## Короткая схема

```text
User
  |
  v
HomeExperience / DecisionConsole
  |  detectInputLanguage()
  |  recentContextTurns()
  |  buildConversationMemory()
  v
POST /api/solve
  |  readLanguage()
  |  readConversationHistory()
  |  readConversationMemory()
  |  getDecisionHistory()
  |  build memory/profile/orchestration instructions
  v
lib/engine.ts
  |  streamingSolveDecision()
  |  OpenAI chat.completions stream
  v
text/plain stream
  |
  v
HomeExperience reads chunks
  |
  v
assistant message appears in UI
  |
  v
solveos.conversation.v1 in localStorage
```

## Основной поток

1. Пользователь пишет сообщение в чате.
2. `DecisionConsole` передает текст в `HomeExperience`.
3. `HomeExperience` сразу добавляет user-turn в состояние текущего чата.
4. `HomeExperience` определяет язык через `detectInputLanguage(message, settings.language.uiLanguage)`.
5. `HomeExperience` собирает тело `SolveRequest`:
   - `problem`
   - `language`
   - `mode`
   - `conversationHistory` из последних неошибочных сообщений
   - `conversationMemory` из локального разговора
   - `streaming: true` для основного чата
6. `HomeExperience` отправляет `POST /api/solve` с `cache: 'no-store'`.
7. `/api/solve` читает JSON body и нормализует вход.
8. `/api/solve` проверяет, что `problem` не пустой.
9. `/api/solve` определяет язык через `readLanguage`.
10. `/api/solve` нормализует историю через `readConversationHistory`.
11. `/api/solve` нормализует клиентскую conversational memory через `readConversationMemory`.
12. `/api/solve` читает серверную историю решений через `getDecisionHistory`.
13. `/api/solve` строит инструкции памяти, профиля, давления, намерения, доверия, сдержанности, энергии, оркестрации и синтеза ответа.
14. Если включен streaming, `/api/solve` вызывает `streamingSolveDecision`.
15. `streamingSolveDecision` собирает system prompt и streaming prompt.
16. OpenAI provider вызывается через `chat.completions.create` с `stream: true`.
17. `/api/solve` возвращает `Response` с `Content-Type: text/plain`.
18. `HomeExperience` читает `response.body.getReader()`.
19. Каждый chunk декодируется через `TextDecoder` и добавляется в assistant-turn.
20. После завершения потока UI снимает `streaming` и `loading`.
21. Эффект сохранения чата записывает компактную историю в `localStorage` по ключу `solveos.conversation.v1`.

## Язык

Клиент первым определяет язык сообщения через `lib/i18n.ts`.

Сервер не должен слепо доверять английскому fallback клиента. Если клиент прислал `en`, `auto` или пустой язык, `/api/solve` повторно определяет язык из текста пользователя через `detectInputLanguage(problem)`.

Ответ и fallback должны использовать язык пользователя, когда это возможно. Если язык не распознан, применяется безопасный fallback.

## Память и история

Есть два разных слоя:

- локальная история чата в браузере
- серверная история решений

Локальная история чата:

- читается из `solveos.conversation.v1`
- хранит последние сообщения текущего разговора
- используется для коротких уточнений и conversational memory
- не является постоянной серверной историей

Серверная история решений:

- читается через `getDecisionHistory`
- используется для memory intelligence, calibration, outcome learning и longitudinal memory
- в non-streaming пути новое решение сохраняется через `saveDecision`

В текущем основном streaming-пути UI получает текстовый ответ, а не `SolveResponse` с `decisionId`. Поэтому спецификация не утверждает, что каждый streaming-ответ создает запись решения на сервере. Локальная история чата при этом сохраняется в браузере.

## Подготовка ответа

`/api/solve` не должен просто переслать пользовательский текст в модель. Перед вызовом provider он собирает контекст:

- нормализованный текст задачи
- язык
- режим
- последние сообщения разговора
- client-side conversational memory
- серверную историю решений
- memory decay
- trust calibration
- restraint и energy signals
- pressure directive
- intent routing
- orchestration instruction
- response synthesis instruction
- профиль пользователя, если он передан

Эти инструкции должны помогать ответу, но не должны показываться пользователю как внутренний текст.

## OpenAI provider

OpenAI provider находится в `lib/engine.ts`.

Основной streaming-путь использует `streamingSolveDecision`:

- нормализует язык
- строит advisor system prompt
- строит streaming answer prompt
- вызывает `getOpenAIClient().chat.completions.create`
- передает `stream: true`
- возвращает `ReadableStream<Uint8Array>`

Если `OPENAI_API_KEY` отсутствует или provider падает до создания потока, `streamingSolveDecision` возвращает локализованный streaming fallback. Если поток прерывается до первого содержимого, он также отдает локализованный fallback.

## Возврат ответа в UI

Основной чат ожидает поток текста.

UI не ждет полного JSON, а постепенно читает stream:

- первый chunk создает assistant-turn
- следующие chunks обновляют последний assistant-turn
- пустой итог считается ошибкой
- ошибка добавляется как assistant-turn с `isError: true`

Сохранение локальной истории происходит отдельно от чтения stream: при изменении thread компактная версия записывается в `localStorage`.

## Non-streaming путь

Non-streaming путь используется, например, для глубокого анализа.

В этом режиме `/api/solve` вызывает `solveDecision`, нормализует `DecisionBlueprint`, применяет routing и calibration, сохраняет решение через `saveDecision`, затем возвращает `SolveResponse` как JSON.

Основной документ про режим глубокого анализа: `спецификации/функции/F004-режим-глубокого-анализа/спецификация.md`.

## Обработка ошибок

Клиент:

- не отправляет пустой ввод на уровне UI
- показывает ошибку API как сообщение в чате
- не очищает текущий разговор из-за ошибки
- завершает `loading` и `streaming` в `finally`
- обрабатывает отсутствие `response.body`
- обрабатывает не-JSON ошибку через fallback copy

Сервер:

- возвращает `400`, если `problem` пустой
- ловит невалидный JSON body
- нормализует поврежденную или слишком длинную историю
- продолжает без memory enrichment, если память не прочиталась
- возвращает локализованную ошибку на `500`
- не должен раскрывать внутренние инструкции

Provider:

- при ошибке OpenAI до начала stream возвращает локализованный fallback
- при прерывании stream до первого текста возвращает локализованный fallback
- в non-streaming пути может перейти в demo/mock fallback через `getMockBlueprint`

## Fallback behavior

Fallback должен быть практичным и коротким. Он не должен притворяться, что была проведена полная модельная оценка, если provider недоступен.

Допустимые fallback-уровни:

- локализованное сообщение ошибки для API/UI
- локализованный streaming fallback в `lib/engine.ts`
- mock blueprint для non-streaming пути
- продолжение без memory enrichment, если память временно недоступна

Fallback не должен:

- очищать историю пользователя
- ломать страницу
- показывать stack trace
- показывать prompt или system instructions
- добавлять новые требования в обход feature specs

## Failure points

- пользователь отправляет пустой текст
- `localStorage` недоступен или содержит поврежденный JSON
- клиент не может определить язык
- `fetch('/api/solve')` возвращает non-OK статус
- у `Response` нет body
- stream возвращает пустой текст
- `/api/solve` получает невалидный body
- история разговора слишком длинная или поврежденная
- чтение серверной памяти падает
- memory enrichment падает
- intent routing дает ранний direct response
- OpenAI API key отсутствует
- OpenAI provider возвращает ошибку, quota или 429
- OpenAI stream прерывается
- модель возвращает неполный blueprint в non-streaming пути
- сохранение решения в non-streaming пути падает

## Что не должно происходить

- нельзя менять runtime-код ради этой спецификации
- нельзя превращать основной чат в обычную болтовню
- нельзя терять короткие уточнения из-за отсутствия контекста
- нельзя показывать пользователю внутренние инструкции
- нельзя считать localStorage серверной памятью
- нельзя менять ключ `solveos.conversation.v1` без обновления `T003`
- нельзя обещать, что модель всегда права
- нельзя скрыто добавлять аккаунты, синхронизацию или внешнее хранение
- нельзя ломать fallback при ошибке модели
- нельзя менять API-контракт без обновления `T002`, `F001`, `F003` и связанных UI-файлов

- `T003` управляет localStorage, серверной памятью и совместимостью хранения.
- User stories показывают примеры, но не добавляют требования.
