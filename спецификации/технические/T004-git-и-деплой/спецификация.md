# T004 — Git и деплой

## Текущая архитектура

Проект ведется как обычный Git-репозиторий. Коммиты должны быть короткими и понятными. Для документации используется тип `docs`.

Сборка выполняется через `npm run build`. Перед сборкой запускается `scripts/response-diversity-test.mjs`, затем `next build`.

## Важные файлы

- `package.json`
- `next.config.ts`
- `docs/commit-rules.md`
- `scripts/response-diversity-test.mjs`
- `AGENTS.md`
- `спецификации/AGENTS.md`

## Поток данных

1. Разработчик читает бизнес-контекст и связанные спецификации.
2. Вносит изменение.
3. Запускает `npm run lint`.
4. Запускает `npm run build`.
5. Проверяет `git status`.
6. Делает один логический коммит.

## Риски

- сборка может зависеть от тестов, которые проверяют поведение ответов
- изменение Next.js настроек может повлиять на runtime
- деплой на Vercel не должен полагаться на запись в локальный файл
- случайный коммит runtime-кода вместе с docs усложнит ревью

## Ограничения

- не коммитить лишние файлы
- не смешивать документацию и функциональные изменения без причины
- не менять `next.config.ts` без технической необходимости
- перед кодом читать документацию Next.js из `node_modules/next/dist/docs/`, если меняется Next.js поведение

## Что нельзя менять случайно

- npm scripts
- правила сборки
- правила спецификаций в `AGENTS.md`
- переменные окружения, связанные с памятью и OpenAI
