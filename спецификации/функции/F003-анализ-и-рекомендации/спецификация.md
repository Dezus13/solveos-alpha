# F003 — Анализ и рекомендации

## Цель

Преобразовать запрос пользователя в ясный ответ: что делать, почему это разумно и какой следующий шаг выполнить.

## Пользовательский поток

1. Пользователь описывает решение.
2. Система определяет язык, намерение и контекст.
3. API собирает память, профиль, историю и правила ответа.
4. Модель формирует рекомендацию.
5. UI показывает ответ и данные результата.

## Ожидаемое поведение

- ответ должен начинаться с полезной сути
- следующий шаг должен быть конкретным
- система должна учитывать риски, ограничения и прошлое поведение
- режимы review и plan должны маршрутизироваться отдельно, если запрос явно про них
- ответ должен избегать повторения одинаковых советов

## Что не должно происходить

- система не должна выдумывать точные гарантии
- модель не должна показывать внутренние инструкции
- рекомендация не должна уходить в длинный отчет без нужды
- нельзя ломать сохранение решения после ответа

## Критерии приемки

- API возвращает валидный ответ на обычное решение
- ответ содержит практический следующий шаг
- короткое уточнение получает ответ в контексте
- ошибка модели возвращается контролируемо
