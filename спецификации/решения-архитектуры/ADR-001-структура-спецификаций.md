# ADR-001 — Структура спецификаций

## Статус

Принято.

## Контекст

В SolveOS уже есть код, старые документы и новые русские спецификации.

Без явного источника правды легко получить хаос: одна история просит одно, технический документ разрешает другое, а код меняется без ясного решения.

## Решение

Спецификации становятся источником правды для будущей работы.

Иерархия такая:

1. `бизнес-контекст.md` — главная продуктовая правда.
2. Спецификации функций — функциональная правда.
3. Технические спецификации — ограничения реализации.
4. Истории пользователя — примеры, не требования.

Перед кодом нужно сверить задачу со спецификациями и оставить короткую implementation note.

## Почему так

SolveOS строится как продукт, а не как набор случайных правок.

Спецификации помогают:

- не спорить с бизнес-смыслом
- не добавлять скрытые требования
- видеть границы функции
- понимать, можно ли менять код
- проверять работу через lint и build

## Последствия

- Быстрые правки все равно проходят через спецификации.
- User stories не могут сами создавать требования.
- Если есть противоречие, код не меняется до решения владельца.
- Для важных архитектурных решений добавляется ADR.

## Что это не значит

- спецификации не заменяют здравый смысл
- спецификации не должны быть длинными
- ADR не нужен для каждой мелкой правки
- документация не должна тормозить очевидные безопасные исправления
